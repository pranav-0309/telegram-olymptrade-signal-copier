## 1. Purpose & Scope

M0 is the first milestone of the Telegram → OlympTrade Signal Copier (PRD v0.7). It establishes the project skeleton — build tooling, package layout, linting/typecheck config, Docker deploy shape, and a stub entrypoint — that subsequent milestones (M1–M11) build on.

**M0 ships a runnable, lint-clean, type-checked, Docker-buildable scaffold with no functional behavior.** All real domain code (parser, state machine, broker, scheduler, DB, Telegram, notifications) is deferred to its own milestone per PRD §15.

**In scope for M0 (12 new files: 11 hand-written + 1 generated):**

1. `pyproject.toml` — PEP 621 metadata + `[tool.uv]`, `[tool.ruff]`, `[tool.mypy]`, `[tool.pytest.ini_options]`
2. `uv.lock` — generated by `uv lock`, checked into git
3. `src/signal_copier/__init__.py` — empty
4. `src/signal_copier/__main__.py` — stub entrypoint, prints message, exits 0
5. `migrations/.gitkeep` — empty placeholder
6. `tests/.gitkeep` — empty placeholder
7. `.env.example` — all env vars from PRD §8 with safe placeholder values
8. `Dockerfile` — `FROM python:3.13`, uv-based install, `CMD ["python", "-m", "signal_copier"]`
9. `railway.toml` — `builder = "DOCKERFILE"`, restart policy
10. `.dockerignore` — exclude patterns
11. `.python-version` — pin `3.13`
12. `.pre-commit-config.yaml` — ruff format + ruff check (no mypy)

**Out of scope (carried over unchanged):** `README.md`, `docs/PRD.md`, `docs/tool-idea.md`, `src/olymptrade_ws/`, `.gitignore`, `OlympTradeAPI/`.

## 2. Resolved Decisions (M0-specific)

The following decisions were confirmed during brainstorming on 2026-06-19. The PRD resolves all other architectural questions (R-1 through R-15).

| # | Decision | Rationale |
|---|---|---|
| D-1 | **Build tooling: uv** (lockfile + tool) | Modern Python packaging default as of 2025. `uv pip install -e .` still satisfies the PRD §15 M0 verification (`pip install -e .` works). `uv.lock` provides reproducible installs across local + Railway. |
| D-2 | **Project license: Proprietary (All rights reserved)** | Personal tool. No `LICENSE` file. `pyproject.toml` records `license = { text = "Proprietary" }`. |
| D-3 | **`migrations/` dir: empty `.gitkeep` in M0** | Aligns filesystem with PRD §7 architecture tree from day one. M4 drops `001_initial.sql` into it. |
| D-4 | **Pre-commit: ruff format + ruff check (no mypy)** | mypy is too slow for pre-commit. Devs can opt in via `pre-commit install`. |
| D-5 | **No CI in M0** | Single-dev personal tool. CI can land in M2+ when there's test surface. |
| D-6 | **Docker base image: `python:3.13` (full)** | User chose full Debian image (~900 MB) for simplicity. No musl/openssl issues with telethon/asyncpg. |
| D-7 | **`python -m signal_copier` stub: print + exit 0** | Required so the container stays up after deploy. M1+ replaces `__main__.py` with the real bootstrap. |
| D-8 | **Single `pyproject.toml` for all tool config** | Modern standard. PEP 621 + `[tool.*]` sections. No split `ruff.toml`/`mypy.ini`. |
| D-9 | **Build backend: hatchling** | uv's default. Works out of the box with `src/` layout. |
| D-10 | **`[project.scripts]` entry: `signal-copier = "signal_copier.__main__:main"`** | Provides console-script entry. Free; lets `signal-copier` work alongside `python -m signal_copier`. |

### 4.2 `Dockerfile`

```dockerfile
FROM python:3.13

WORKDIR /app

# Install uv
COPY --from=ghcr.io/astral-sh/uv:latest /uv /uvx /usr/local/bin/

# First layer: copy only what's needed for `uv sync` (cache-friendly).
# README.md is required because pyproject.toml declares `readme = "README.md"`
# and hatchling validates the readme at build time.
COPY pyproject.toml uv.lock README.md ./
COPY src/signal_copier/__init__.py ./src/signal_copier/__init__.py
COPY src/signal_copier/__main__.py ./src/signal_copier/__main__.py

# Install runtime deps only (--no-dev skips pytest/ruff/mypy in the image)
RUN uv sync --frozen --no-dev

# Now copy the rest
COPY src/ ./src/
COPY migrations/ ./migrations/

# Run as non-root
RUN useradd --create-home --shell /bin/bash app && chown -R app:app /app
USER app

# Use the venv's Python explicitly. The system Python doesn't have the package
# installed; the venv created by `uv sync` is at /app/.venv.
CMD ["/app/.venv/bin/python", "-m", "signal_copier"]
```

**Notes:**
- `--frozen` refuses to modify `uv.lock` during build — fails the build if the lockfile is out of sync with `pyproject.toml`. Catches the "I forgot to run `uv lock`" mistake at deploy time.
- `--no-dev` skips dev deps (pytest, ruff, mypy, pre-commit) in the container — smaller image, faster deploys.
- `__main__.py` is included in the first `COPY` step (alongside `__init__.py`) because Hatchling requires at least one non-`__init__.py` Python file in a package to build a wheel. Without it, the install step fails.
- `README.md` is included in the first `COPY` step (alongside `pyproject.toml` and `uv.lock`) because `pyproject.toml` declares `readme = "README.md"` and hatchling validates the readme at build time. Without it, `uv sync` fails with `OSError: Readme file does not exist: README.md`.
- The CMD uses the venv's Python (`/app/.venv/bin/python`) explicitly. The system Python doesn't have `signal_copier` installed — it's in the venv. Without this, the container exits with `No module named signal_copier`.
- No `LICENSE` file is `COPY`-ed (D-2: proprietary, no LICENSE).

## 5. Dependency Policy (staged)

M0 ships with **zero runtime dependencies** and five dev dependencies. Runtime deps are added per milestone as their consumers are implemented.

### 5.1 M0 dev dependencies (in `pyproject.toml`)

| Package | Version | Purpose |
|---|---|---|
| `pytest` | `>=8.3` | Test runner |
| `pytest-asyncio` | `>=0.24` | Async test support (configured, used from M2 onward) |
| `ruff` | `>=0.7` | Linter + formatter |
| `mypy` | `>=1.13` | Strict type checker |
| `pre-commit` | `>=4.0` | Git hook manager |
| `hatchling` | (build backend) | uv's default; powers `[build-system]` |

### 5.2 Runtime dependencies (added per milestone)

| Package | Added at | Why staged |
|---|---|---|
| `pydantic-settings` | M1 | First config layer; no `.env` reading in M0's stub. |
| `loguru` | M1 (or M7) | M0 uses `print()`, not loguru. Loguru arrives with the first real logger setup. |
| `asyncpg` | M4 | No DB code in M0–M3. |
| `telethon` | M5 | First import in `telegram/client.py`. M0's stub doesn't need it. |
| `olymptrade_ws` | **Vendored**, not a dep. Imported via `from olymptrade_ws import …` from the in-tree `src/olymptrade_ws/`. Not declared in `pyproject.toml`. uv's editable install picks it up because it's under `src/`. | M8 |

**Rationale for staging:**
- A M0 with zero runtime deps is honest: nothing in M0 actually uses third-party code, so listing telethon/asyncpg would be aspirational and force `uv sync` to download wheels that aren't exercised.
- Each milestone adds deps alongside the code that uses them, so `uv.lock` always reflects real usage.

### 5.3 Re-vendoring compatibility

When the user re-vendors `olymptrade_ws` (per `src/olymptrade_ws/VENDORED.md`), `pyproject.toml` does not change. The vendored code is consumed via Python's normal `sys.path` resolution from `src/`. If a future restructuring requires an explicit path dep (e.g., `olymptrade_ws @ file://./src/olymptrade_ws`), it's added under `[tool.uv.sources]`. Defer until that need arises.

## 6. Developer Workflow

### 6.1 Local dev

```bash
# 1. Install uv (one-time)
#    Windows: powershell -c "irm https://astral.sh/uv/install.ps1 | iex"
#    macOS/Linux: curl -LsSf https://astral.sh/uv/install.sh | sh

# 2. Clone and sync
git clone <repo-url>
cd olymptrade
uv sync                                  # creates .venv/, installs deps, validates uv.lock

# 3. Verify
uv run pytest                            # "no tests ran" — expected for M0
uv run ruff check .
uv run ruff format --check .
uv run mypy src/signal_copier
uv run python -m signal_copier           # prints "signal_copier M0 scaffold: not implemented yet. See docs/PRD.md."

# 4. (Optional) Wire pre-commit
uv run pre-commit install
# Now `git commit` auto-runs ruff format + ruff check --fix on staged files.
```

### 6.2 Docker build (verifies the deployable shape)

```bash
docker build -t signal-copier-m0 .
docker run --rm signal-copier-m0         # prints the stub message, exits 0
```

### 6.3 Idempotency and determinism

- `uv sync` is deterministic given `uv.lock` (pinned hashes). Two devs running `uv sync` on the same `uv.lock` get byte-identical environments.
- `uv sync --frozen` in the Dockerfile refuses to modify `uv.lock` — fails the build if the lockfile is out of sync with `pyproject.toml`.
- `pytest` with 0 tests exits with code 5 ("no tests collected") by default. The PRD §15 M0 row says "pytest runs (0 tests)" — exit code 5 is the canonical "0 tests" signal. If the user wants exit code 0, an empty `tests/__init__.py` can be added; defer until requested.

### 6.4 Failure modes

| Failure | Surface |
|---|---|
| Wrong Python version | `uv sync` reads `.python-version` and refuses; user runs `uv python install 3.13` |
| `uv.lock` out of sync with `pyproject.toml` | `uv sync` errors with clear message; `uv lock` regenerates it |
| Docker build fails | `docker build` exit code ≠ 0; log shows the failing step |
| Docker run crashes | `docker run` exit code ≠ 0; Railway would mark the deploy as failed and not promote it |
| Missing `__main__.py` in copy step | Hatchling wheel build fails; Dockerfile pre-step catches this |

## 7. Risks & Edge Cases

1. **`uv` is a moving target.** The lockfile format and `pyproject.toml` schema evolve. M0 pins `hatchling` and uses `pyproject.toml` fields supported by `uv>=0.4`. If uv ships a breaking change in M1+, `uv.lock` may need regeneration. Mitigation: pin uv in a tool-versions file (defer; not standard for Python projects).

2. **Hatchling's `src/` discovery requires at least one non-`__init__.py` file.** M0 works around it by including `__main__.py` in the first Docker layer copy. The entrypoint is permanent, so the build will not break from this in future milestones.

3. **`python:3.13` (full) is ~900 MB.** User explicitly chose this for simplicity. Railway Hobby plan has 32 GB outbound bandwidth/month and ample disk; this is acceptable. The image is rebuilt only on `pyproject.toml`/`uv.lock` change (cache layer) or on base-image update (rare).

4. **No tests in M0.** PRD §15 M0 row says "pytest runs (0 tests)" — that's the explicit acceptance. The risk of "M0 ships and people merge broken code" is mitigated by the developer running `uv run pytest` + `ruff` + `mypy` locally (no CI yet, per D-5).

5. **The stub prints to stdout, not via loguru.** PRD §4.7 FR-7.2 says "Mirror every DM at INFO level with same payload" via loguru. M0's stub has no loguru setup. M1 (or M7) will add loguru and replace `print()` with `logger.info()`. Documented transition point.

6. **No `OLYMP_ACCOUNT_GROUP=demo` guardrail in M0.** PRD §4.6 FR-6.6 says the app refuses to start with `OLYMP_ACCOUNT_GROUP=real` while `DRY_RUN=false`. M0's stub doesn't read config. M0 ships `.env.example` with `OLYMP_ACCOUNT_GROUP=demo` and a comment about the guardrail landing in M1. **Acceptable: the stub doesn't place trades.**

7. **Vendored package is not declared in `pyproject.toml`.** It works because `src/` is on `sys.path` via editable install, but a strict checker might warn. We accept this — the vendored layer is intentionally separate (per R-15, see PRD §13.1).

8. **`.env` is in `.gitignore` (existing rule).** `.env.example` is the only env file tracked. User must copy `.env.example` → `.env` and fill in real values locally. Documented in the existing README.

9. **M0 is platform-agnostic.** The Dockerfile runs on Linux (Railway). The dev workflow runs on Windows / macOS / Linux (uv is cross-platform). No platform-specific code in M0.
